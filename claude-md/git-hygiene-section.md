# Git hygiene (all repos under ~/src)

Full playbook: `~/.claude/docs/git-hygiene-playbook.md`. The rules, uniform for every repo with a branch literally named `main` or `master` (a differently named default branch is unguarded):

1. **Always worktree.** Every change — feature, fix, typo — happens on a branch in a herdr worktree, never in the main checkout. Hooks catch the common accidental slips: file edits in a primary checkout (any branch — `git checkout -b` there is not the worktree flow), direct commits on main outside a squash-merge landing, and branch commits in a primary checkout. This rule is absolute, no exceptions. When a hook denies you, create a worktree and route the work per rule 2 below — do not look for workarounds. (Bootstrapping a brand-new repo is the one exception the hooks allow — the first commit on a repo with no commits yet is fine.)
2. **Route who implements on discovery and volume, not on importance or diff size.** The worktree from rule 1 is non-negotiable; who executes inside it is not. Pick the cheapest tier that fits:
   - **Tier 1 — self-edit.** You already know the exact change, no discovery needed: create the worktree, `EnterWorktree` into it, edit and commit directly, `ExitWorktree(action: "keep")`, land with `bin/land.sh`. Works because the edit-guard hook is cwd-based, not agent-identity-based — any agent whose cwd is inside the worktree may edit it.
   - **Tier 2 — in-session subagent, worktree-isolated.** Needs discovery, volume, or repetition but modest blast radius: Claude Code's Agent tool with `model: haiku|sonnet` and `isolation: "worktree"` pins the subagent's cwd inside its own worktree, so the same cwd-based guard permits its edits. One tool call, no watcher. **Unvalidated in remote-less repos** — `isolation: "worktree"`'s default `baseRef` branches from `origin/<default-branch>`, untested with no `origin`; it also creates a Claude-native worktree under `.claude/worktrees/`, invisible to `herdr worktree list`.
   - **Tier 3 — dispatch a herdr pane worker.** Real blast radius, a long horizon, or parallel work: the session in the main checkout is the ORCHESTRATOR — it launches a Claude agent inside the worktree workspace and supervises, and does NOT edit files itself, not even at the worktree's path. With `HERDR_ENV=1`:
     ```bash
     herdr worktree create --cwd <repo> --branch <feat|fix|chore>/<slug> --no-focus --json
     # parse .result.root_pane.pane_id
     herdr pane run <pane> "claude"          # or claude --model <m> — size by task, see below
     herdr wait agent-status <pane> --status idle --timeout 30000
     herdr pane run <pane> "<full self-contained task prompt>"
     ```
     **Size the worker (model + effort) to the task's hardest requirement** — dispatch the cheapest model that clears it, reserve your top-tier model for the genuinely hard tail, and pass `--model`/`--effort` explicitly (your harness default is not the right dispatch default). Add your own model-routing policy here if you have one. If a worker grinds without progress, kill it and relaunch on a stronger combo — workers don't self-escalate. The dispatched model is the worker's *ceiling*, not what it uses for everything: it keeps planning/design/judgment on its own model and pushes routine subtasks to cheaper subagents.
     **Feature lifecycle — one worktree, one worker, one landing.** The orchestrator's pre-dispatch role is SCOPING ONLY: ask the user whatever clarifying questions you need to pin down the goal, constraints, and blast radius — asking is expected, not a delay, since a well-scoped brief is what makes dispatch smart — then use the answers to size the model/effort dials and write a complete self-contained brief, and dispatch ONE worker. Clarify only enough to dispatch well; the design itself still happens in the worker's pane, not here. The worker owns the full lifecycle in its worktree — brainstorming/design (the user interacts with it directly in its pane when design input is needed; `blocked` status signals this), spec (first commit), plan, implementation, tests — landed in a single squash. The orchestrator does not brainstorm designs, author specs or plans, or dispatch workers just to commit documents; it scopes, dispatches, supervises, and lands. In a main checkout, "let's brainstorm/design/build/implement X" is a DISPATCH trigger, NOT an inline-work trigger — do not invoke brainstorming/spec/plan skills inline; create the worktree and let the worker invoke them in its pane (a `UserPromptSubmit` hook nudges this).
     Then confirm the worker started (`herdr wait agent-status <pane> --status working --timeout 30000`) and supervise WITHOUT blocking: run `bash ~/.claude/bin/herdr-watch-agent.sh <pane>` as a background Bash task (`run_in_background: true`) and END THE TURN — never foreground-wait on a worker. This keeps the orchestrator free to talk to the user and dispatch more worktree workers in parallel (one watcher per pane). The watcher wakes the session when it exits: `done`/`idle` → review with `herdr pane read <pane> --source recent-unwrapped --lines 120`, then the orchestrator runs `/land` (never the worktree agent — landing removes its workspace); `blocked` → read the question, answer via `pane run`, restart the watcher; `gone` → pane closed, investigate. Without herdr, fall back to doing the work yourself in a plain `git worktree add` checkout.
3. **Land via `bin/land.sh <branch>`.** It scripts rebase on main → tests (or build/lint if no suite) → prints the diff for review → race-check → `git merge --squash` + `git commit` on main. Read the diff it prints (or pass `--dry-run` to stop there) before trusting the merge. It refuses to land main onto itself, a dirty branch worktree, or tracked changes in the main checkout (untracked ones are reported, not fatal).
4. **Cleanup is automatic on landing** — `bin/land.sh` removes the worktree and deletes the branch as its last step. Abandoning a branch instead: `herdr worktree remove --workspace <id> --force` then `git branch -D <branch>`.
5. **Deploy is separate.** Merging makes main deployable; never auto-deploy after a merge — deploy only when explicitly asked.
6. Non-git directories are exempt.

**What this does and does not enforce.** The hooks are guardrails against a careless agent, not a security boundary against an adversarial one, and catch a direct `git commit` on main and a direct edit to the primary checkout, pre-execution. The commit guard parses the `Bash` command string, so a command routed through another interpreter (`sh -c "..."`) or a non-`commit` git operation (`merge`, `pull`, `cherry-pick`, plumbing) isn't recognized. The edit guard resolves the target via git rather than parsing text, but only sees `Edit`/`Write`/`NotebookEdit` — any other write path (an MCP server's shell/file tools, `sed -i`, `patch`) never reaches it, and it also exempts `~/.claude/**`, where the hooks themselves live, so an `Edit` there can turn enforcement off in the same session. Neither governs your own terminal. Full breakdown, including the hard-enforcement options this repo doesn't implement (git-level hooks, remote branch protection): the playbook.
